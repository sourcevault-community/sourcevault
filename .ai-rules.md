# AI Assistant Rules

These rules apply to all AI coding assistants (e.g., Gemini, Claude, Agent, CLI) interacting with this repository. 
You MUST follow these rules during every interaction.

## 1. Commit Hygiene
- **Never leave uncommitted work**: After successfully implementing a feature, fixing a bug, or making a requested change, you MUST stage and commit your changes using `git add` and `git commit`.
- **Pre-flight Checks**: Before committing, always run `git status` or `git diff --staged` to ensure you are only committing the intended files.
- **Commit Messages**: Write descriptive, atomic commit messages. If a task involves multiple logical parts, create separate commits. 

## 2. Changelog Maintenance
- **Keep a Changelog**: This project maintains a `CHANGELOG.md` file following the [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) format.
- **Update Unreleased**: Any time you add a feature, fix a bug, or make notable changes, you MUST add an entry to the `## [Unreleased]` section of the `CHANGELOG.md` file.
- **Categorization**: Place your changelog entries under the appropriate sub-header (e.g., `### Added`, `### Changed`, `### Deprecated`, `### Removed`, `### Fixed`, `### Security`). If the sub-header does not exist under `[Unreleased]`, create it.

## 3. Source Code Standards
- **Standard Header**: All new source code files MUST include the standard license header at the top of the file. The exact content for this header can be found in the `HEADER` file at the root of the repository.

## 4. Testing & Verification
- **Always Test and Build**: Before committing any new code or modifications, you MUST verify the source has no errors. This involves running the appropriate build commands (e.g., `make build` or `go build ./...`) and testing commands (e.g., `make test` or `go test ./...`) to ensure everything compiles and runs successfully. Never commit broken code.

## 5. Workflow Example
When the user asks for a change, your workflow should look like this:
1. Make the code changes.
2. Run build and tests to ensure no errors.
3. Update `CHANGELOG.md` with the description of the change under `[Unreleased]`.
4. Verify the changes work (if applicable).
5. Run `git add <files>` including `CHANGELOG.md`.
6. Run `git commit -m "feat/fix: <description>"`.
7. Inform the user of the successful completion and commit.
